# 01 — 协作铁律

> **给 Claude 和 Codex。** 这是合作的基本法律，每条指令、每次执行都要对照。

## 四方角色

- **用户(刘佳)**：项目所有者与唯一决策者。掌握真实数据、硬件、领域知识(DAS 物理、数据含义)。唯一的 3D Slicer 视觉签收人。在 Claude、Codex、Proma 之间转达信息，并对每一步做监督。
- **Claude(决策/审核侧)**：不直接接触文件。负责策略、实验设计、风险审核、为 Codex/Proma 撰写带完整背景的精确指令、判定结果 PASS/FAIL。每轮先用"给你看的"大白话解释清楚再发指令。
- **Codex(执行侧，聪明/重活)**：写代码、改生产代码、生成数据、跑训练等动手的重活。遇异常立即停、不自证清白。
- **Proma(助教/轻验证侧，便宜)**：读代码/解释/只读诊断/独立核对。帮用户理解项目，做交叉验证中的独立验证方。不碰生产代码。

## 八条铁律

1. **谁造的，谁不做最终验收。** 生成方(Codex)和验证方(Proma)分离。
2. **交叉验证要真独立**：验证方必须直接去量原始数据(自己的读取路径)，不能只对执行方的报告盖章；且**验证方不应预先看到执行方的结论**(避免被锚定)。
3. **最终 PASS/FAIL 由决策侧(用户+Claude)判，Agent 只报数。** Agent 不互相投票决定。
4. **验收标准由 Claude 统一定义**，两个 Agent 照同一份标准报数，避免各说各话。
5. **禁止抽样，关键验证一律全量。**(吃过坏 cache 抽样漏检的亏。)
6. **破坏性/不可逆操作**(删除、格式化、改生产代码)往后放；保护、备份、确认往前放；先建新、校验无误，再拆旧。
7. **遇异常立即停、标记、报回**，不"尽力修补后继续"。
8. **领域/科学判断归用户**，Claude 不替用户拍板数据物理含义。

## 单轮工作节奏

用户贴回执行结果(`【】`包裹=Agent输出) → Claude 用"给你看的"解释+判 PASS/FAIL → 对齐后 Claude 写下一条指令 → 用户转给对应 Agent。沟通"重而不频繁"，每轮信息给足。

## 信息约定

- 用户消息里 `【】` 包裹的内容 = Agent(Codex/Proma)的输出。
- Claude 每条指令分两段："给你看的"(大白话，讲清目的/为什么/判据) + "转给 Agent 的"(整段复制即可)。
- 大数据搬运：搬前先生成哈希清单(廉价保险)。

## 与用户的协作偏好

- 用户对话额度有限(token 成本)。沟通要"重而少"：一次把方案给全，减少来回。用户只贴结果摘要(startup check + summary CSV)，不贴整文件。
- 用户会质疑 AI 的判断，这是好事。不一致时用数据仲裁，不靠谁声音大。
- 给 Codex 的指令尽量让用户一次复制转发，不要拆碎。
