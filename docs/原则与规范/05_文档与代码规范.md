# 05 — 文档与代码规范

> **给 Claude、Codex、Proma。** 怎么记录、怎么写文档、怎么组织代码。

## 发现即记录

任何有意义的发现、诊断结论、设计决策，**发现后立即写入对应文档**，不等事后补。

**什么需要记录**：管线级发现（fp16 溢出）、设计决策（选 scale 而非 clip）、背景知识（为什么 phantom 零溢出）、踩过的坑（抽样漏检、脚本语法错）

记录格式：**时间 + 管线阶段 + 影响谁 + 写到哪**

| 发现类型 | 更新文档 |
|----------|----------|
| 管线/数据/I/O 问题 | `项目笔记/02_数据管线详解.md` |
| Codex 执行约束 | `Codex参考/` |
| Claude 战略信息 | `Claude参考/` |
| 协作流程问题 | `原则与规范/` |

## 文档语言原则

**所有文档用中文书写，方便用户阅读。这是最高优先级。**

1. 文件名用中文，让人一眼知道内容。
2. 正文用中文。技术术语保留英文（DAS、fp16、memmap），解释用中文。
3. 不维护英文平行版本。论文需要时再单独整理。
4. 代码注释、变量名、config 键名可以保持英文，但文档中代码块说明用中文。

> 违反自查：用户看到文件名/段落，能否不靠任何人解释直接读懂？不能就改。

---

### 命名铁律（Proma 必守）

任何新建文件或文件夹，命名必须让人一眼看懂。违反即返工。

**正面示例**：
-  — 有序号，有内容描述
-  — 项目名 + 用途后缀
-  — 英文但自解释

**反面示例（禁止）**：
-  — 下划线开头，意义不明
- 、 — 代号而非名称
-  — 缩写无上下文

**自检**：如果你需要打开文件才知道里面是什么，名字就不合格。


## 单事实源原则

一个事实只在一个文档里写完整版本，其他文档只放引用。

1. **写入前先判断主文档**：管线事实 → `项目笔记/02`；协作纪律 → `原则与规范/`；Codex 约束 → `Codex参考/`；实验记录 → `Claude参考/`
2. **其他文档只引用不复制**。例：Codex参考里写 `cache 须 normalize=True，详见 项目笔记/02_数据管线详解.md`
3. **更新只改主文档**，引用方不动。

> 为什么：避免同一事实多个版本不一致，不同角色基于不同信息做判断。

## 代码组织原则

项目代码按角色分目录，根目录只有生产代码和子目录。

```
RF_Image_GAN/
├── *.py                   ← 生产代码
├── scripts/               ← 一次性脚本（diagnostics/verification/migration）
├── tests/                 ← 可重复测试
├── experiments/           ← 实验运行
└── docs/                  ← 文档
```

铁律：根目录不放一次性脚本、脚本用完即归档至 scripts/、新脚本先点火测试再全量跑。

---

## 大规模数据处理铁律

处理超过 100 个文件或预计耗时超 5 分钟的任务：

1. **点火测试**：先拿 3~5 个文件验证脚本正确（语法/NaN/shape）
2. **全量执行**：通过后全量跑，输出进度到日志文件
3. **完整性校验**：跑完后确认数量、无遗漏
